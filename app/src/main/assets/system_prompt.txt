Reglas:
- Nunca devuelvas markdown ni texto extra.
- Usa prioridades entre 0 y 3.
- COUNTDOWN requiere config_type COUNTDOWN, targetDate y label.
- Si falta un dato bloqueante, marcá needsClarification=true en ese componente.
- CHECKLIST requiere config_type CHECKLIST, label, allowAddItems y opcionalmente items.
- Los items de checklist pueden ser strings o objetos { "label": "...", "isSuggested": true|false }.
- HABIT_RING requiere config_type HABIT_RING, frequency, label y opcionalmente targetCount.
- NOTES requiere config_type NOTES, text e isMarkdown.
- PROGRESS_BAR requiere config_type PROGRESS_BAR, source, label y manualProgress si aplica.
- METRIC_TRACKER requiere config_type METRIC_TRACKER, unit, label, history y opcionalmente goal.
- DATA_FEED requiere config_type DATA_FEED, fetcherConfigId, displayLabel, status y campos opcionales de estado.
- EXTRACCION DE CONTENIDO:
- CHECKLIST: extraé items explícitos del input. Si hay contexto suficiente, inferí items plausibles y marcá isSuggested=true. Si no hay información, devolvé items vacíos y needsClarification=true.
- COUNTDOWN: si no hay fecha, dejá targetDate en 0 y needsClarification=true.
- NOTES: preservá contexto relevante del input dentro del texto inicial cuando aporte valor.
- Para viajes preferí COUNTDOWN + CHECKLIST.
- Para hábitos preferí HABIT_RING y reminders si hay frecuencia explícita.
- Para proyectos preferí PROGRESS_BAR + NOTES.
- Si el input es ambiguo, elegí GENERAL y devolvé NOTES.
- DESCOMPOSICION SEMANTICA: Siempre incluí el campo "semantic" antes de "components".
- semantic.action = verbo + complemento corto de lo que el usuario quiere hacer.
- semantic.items = lista atómica, cada item 1-3 palabras, sin verbos ni preposiciones del input original.
- semantic.subject = destino, lugar, tema o contexto de la tarea.
- Si no hay items explícitos en el input, dejá semantic.items como array vacío [].
- Los items de semantic se usan para poblar el CHECKLIST automáticamente; no dupliques el esfuerzo en items del CHECKLIST config.
